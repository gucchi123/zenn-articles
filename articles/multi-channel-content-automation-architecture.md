個人プロジェクトとして、動画・記事コンテンツを **16以上のプラットフォーム** に自動配信する基盤を作って運用しています。単純に「同じ内容を全部に流す」だけなら大したことはないのですが、実際にやってみると**プラットフォームごとに前提がまるで違う**ことが最大の難所でした。この記事では、その設計でぶつかった壁と、媒体の性質別にどう対応を分けたかをまとめます。

## 前提: 「同じ内容を全部に流す」が破綻する理由

最初は「1本のコンテンツを作って、全プラットフォームに同じように配信するスクリプトを1本書けばいい」と考えていました。すぐに破綻しました。理由は大きく3つです。

1. **公式APIの有無・機能差がプラットフォームごとに全然違う**
2. **自動投稿そのものがBANリスクになるプラットフォームがある**
3. **新規アカウントは最初から全力運用すると凍結される**

これらを無視して機械的に量産すると、たった数週間で複数アカウントが止まる/凍結されるという経験をしました。以降は「媒体の性質を先に分類してから、対応を設計する」という順番に変えています。

## 媒体を4つの軸で分類する

現在は、新しいプラットフォームを追加するとき、以下の軸でまず分類しています。

### 軸1: 公式APIで投稿できるか

投稿系のAPIが公式に存在するプラットフォームは自動化しやすい一方、**実は投稿できてもある種の機能だけAPIが存在しない**、というケースに何度も遭遇しました。たとえば動画共有系のあるプラットフォームでは、通常の投稿APIはあるのに「コミュニティ投稿」的な副次機能だけAPIが提供されていない、といった具合です。こうした欠落に気づかずに「なぜかこの機能だけ反映されない」を長時間デバッグしたことがあります。

対応方針: **APIが無い機能は、下書きを自動生成してメッセージアプリ（Teams等）に通知し、人間が数十秒でコピペ投稿する**、という半自動パターンに落ち着きました。全自動化にこだわらず、「機械が作る・人が押す」の分業にすることで、公式APIの欠落に振り回されずに済みます。

```python
# 下書き生成 → 通知 → 人間が最後のワンタップ、というパターンの骨格
def build_draft(metadata: dict) -> str:
    caption = build_caption(metadata)
    save_to_local(caption)
    return caption

def notify_teams(draft_text: str):
    # チャットに下書きを送るだけ。実際の投稿ボタンは人間が押す
    ...
```

### 軸2: 自動投稿がBANリスクになるか

同じ「自動投稿」でも、プラットフォームによってポリシー上の扱いが全く違います。ある記事投稿系プラットフォームでは、実際に**「事業者による宣伝・勧誘の反復」を明示的に禁止**しており、これはリンクの有無を問わない広い禁止規定でした。別のプラットフォームでは、逆に商用利用・アフィリエイトは公式に許可されている一方で「自動生成コンテンツ」「大量重複投稿」を禁止事項として明記していました。

これを事前に調べずに「まあ自動化しておけば楽だろう」で進めた結果、あるプラットフォームでアカウントが凍結される事故を経験しました。以降は、**新しいプラットフォームを追加する前に必ず利用規約の禁止事項を確認し、機械的な反復投稿がグレーな媒体は「人間が手動で、都度言い換えて投稿する」運用に倒す**、というルールを徹底しています。

### 軸3: 新規アカウントの立ち上がり期をどう扱うか

新規開設したアカウントで初日からフル頻度で投稿すると、スパム判定される確率が明確に上がります。これも実際に一度、新規開設した記事投稿サービスで痛い目を見ました。

対応方針として、**新規アカウントは最初の2週間は意図的に低頻度・高品質に絞り、実績ができてから徐々に頻度を上げる**というフェーズ管理を導入しました。既存の失敗事例をルールとして明文化しておくことで、次に新しい媒体を追加するときに同じ轍を踏まなくなります。

### 軸4: コンテンツフォーマットの相互流用性

動画系プラットフォームは、実は素材の使い回しが効きやすいことに気づきました。1本の縦型ショート動画を作れば、複数の動画プラットフォームにほぼそのまま流用できます。これに気づいてからは、「新しい動画系プラットフォームを追加するコスト」が劇的に下がりました。既存の動画資産を流用するだけで、追加の制作コストほぼゼロで新しい配信面を増やせます。

## 「静かな失敗」への対策が一番効いた

運用していて一番怖かったのは、エラーは出ないのに「実は動いていない」パターンでした。具体的には:

- ネタが尽きたバッチ処理が、**exit code 0 (正常終了) のまま何も生成せずに終わり続ける**
- あるプラットフォーム向けの投稿ロジックが、メタデータのキー名の想定違いで**特定のコンテンツ形式だけ静かにスキップし続ける**
- 通知ロジックのバグで、**失敗しても誰にも通知が飛ばない**

これらは全て「ログを能動的に見に行かないと気づけない」種類のバグで、実際に数日〜数週間気づかずに放置していたケースがありました。

対策として、今は**「正常に見える状態」と「本当に正常な状態」を区別するチェック層**を別立てで用意しています。具体的には:

1. 各投稿処理は必ず戻り値（成功/失敗）を確認し、無条件の成功表示をしない
2. コンテンツの「在庫」を扱うバッチは、在庫切れ自体を異常として通知する（正常終了として握りつぶさない）
3. 全ての失敗・警告は、人間が日常的に見る通知チャンネルに必ず流す

```powershell
# NGパターン: 戻り値を見ずに成功扱いしてしまう
foreach ($poster in $posters) {
    & $poster
    Write-Output "distributed: $poster"  # 実際は失敗していても表示される
}

# OKパターン: 戻り値を見て、失敗を握りつぶさない
foreach ($poster in $posters) {
    $result = & $poster
    if ($LASTEXITCODE -eq 0) {
        Write-Output "OK: $poster"
    } else {
        Write-Output "FAIL: $poster"
        # ここで初めて気づける
    }
}
```

## まとめ

複数プラットフォームへの自動配信基盤は、「投稿ロジックを1本書けば終わり」ではなく、**媒体ごとの制約（API有無・BANリスク・新規アカウントの扱い・フォーマット互換性）を先に分類し、それぞれに応じた設計をする**ことが本質でした。特に「エラーは出ないが実は動いていない」系の静かな失敗への対策は、後から入れるほど発見が遅れるコストが大きいので、最初から仕組みに組み込んでおく価値があると感じています。
